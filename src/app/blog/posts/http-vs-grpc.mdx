## Overview

When building distributed systems, the communication protocol between services is one of the most foundational decisions you'll make. For years, **HTTP/REST** has been the dominant choice — it's simple, well-understood, and works everywhere. But as your system scales and your services multiply, the cracks begin to show.

In the **G-NEXA** project — an enterprise polyglot microservices system spanning Go, NestJS, and Express — we started with HTTP for all inter-service communication. As the architecture grew to 5+ independent services, we hit critical bottlenecks in latency, payload size, and type safety. This led us to migrate service-to-service communication to **gRPC**.

This article covers the technical reasoning behind that decision and the lessons learned along the way.

## What is gRPC?

**gRPC** (gRPC Remote Procedure Calls) is a high-performance, open-source RPC framework originally developed by Google. Unlike REST, which operates on resources and HTTP verbs, gRPC operates on **method calls** — you define services and methods in a `.proto` file, and the framework generates client and server code for you.

```protobuf
syntax = "proto3";

service ProductService {
  rpc GetProduct (ProductRequest) returns (ProductResponse);
  rpc ListProducts (ListRequest) returns (stream ProductResponse);
}

message ProductRequest {
  string id = 1;
}

message ProductResponse {
  string id = 1;
  string name = 2;
  double price = 3;
  int32 stock = 4;
}
```

## HTTP/REST vs gRPC: The Key Differences

### 1. Serialization Format

REST typically uses **JSON** — human-readable but verbose. gRPC uses **Protocol Buffers** (protobuf), a binary serialization format that is significantly smaller and faster to parse.

```json
{
  "id": "prod_001",
  "name": "Wireless Headphones",
  "price": 79.99,
  "stock": 142,
  "category": "Electronics",
  "tags": ["audio", "wireless", "bluetooth"]
}
```

The same data in protobuf is roughly **60-80% smaller** in binary representation, and parsing is **5-10x faster** than JSON.

### 2. Transport Protocol

REST runs on **HTTP/1.1** (though HTTP/2 is possible, it's rarely the default). gRPC **requires HTTP/2**, which gives it:

- **Multiplexing**: Multiple requests over a single TCP connection
- **Header compression**: Reduced overhead via HPACK
- **Binary framing**: More efficient data transfer

### 3. Type Safety & Code Generation

With REST, your API contract lives in documentation (OpenAPI/Swagger at best). With gRPC, the `.proto` file **is** the contract, and the framework generates strongly-typed client and server stubs:

```bash
# Generate Go server and client code
protoc --go_out=. --go-grpc_out=. proto/product.proto

# Generate TypeScript client for NestJS
protoc --plugin=protoc-gen-ts_proto \
  --ts_proto_out=./src/proto \
  proto/product.proto
```

This means compile-time type checking across **different programming languages** — a game-changer for polyglot architectures.

### 4. Streaming

REST is inherently request-response. gRPC supports **four communication patterns**:

- **Unary**: Traditional request-response (like REST)
- **Server streaming**: Server sends a stream of responses
- **Client streaming**: Client sends a stream of requests
- **Bidirectional streaming**: Both sides stream simultaneously

```protobuf
service OrderService {
  // Unary
  rpc GetOrder (OrderRequest) returns (OrderResponse);
  
  // Server streaming - real-time order updates
  rpc WatchOrder (OrderRequest) returns (stream OrderUpdate);
  
  // Client streaming - batch order creation
  rpc CreateOrders (stream CreateOrderRequest) returns (BatchResponse);
  
  // Bidirectional - live chat between services
  rpc SyncInventory (stream InventoryUpdate) returns (stream InventoryStatus);
}
```

## Why We Migrated: The G-NEXA Case Study

In G-NEXA, our architecture consists of:

- **Product Service** (Go) — Catalog and inventory
- **Order Service** (NestJS) — Orders, invoices, cart management
- **User Service** (NestJS) — Authentication and store profiles
- **Finance Service** (Express) — Payment processing
- **Media Service** (Express) — File uploads and CDN management

When the Order Service needs to validate a product and check stock before creating an order, it must call the Product Service. With REST, this looked like:

```typescript
// Old HTTP approach in Order Service (NestJS)
async validateProduct(productId: string): Promise<Product> {
  const response = await this.httpService.get(
    `http://product-service:8080/api/products/${productId}`
  );
  return response.data; // No type safety — just trust the shape
}
```

The problems we encountered:

1. **Latency**: JSON serialization/deserialization added measurable overhead at scale
2. **No type safety**: A field rename in the Go Product Service would silently break the NestJS Order Service
3. **Boilerplate**: Every service needed custom HTTP clients with error handling, retry logic, and timeout configuration
4. **No streaming**: Real-time inventory sync required separate WebSocket infrastructure

After migrating to gRPC:

```typescript
// New gRPC approach in Order Service (NestJS)
@GrpcMethod('ProductService', 'GetProduct')
async validateProduct(productId: string): Promise<ProductResponse> {
  return this.productClient.getProduct({ id: productId }).toPromise();
  // Strongly typed — breaks at compile time if contract changes
}
```

## Performance Results

After migrating, our internal benchmarks showed:

| Metric | HTTP/REST | gRPC | Improvement |
|--------|-----------|------|-------------|
| Avg latency (p50) | 12ms | 3ms | **4x faster** |
| Payload size | 2.4KB | 0.8KB | **67% smaller** |
| Throughput (req/s) | 8,200 | 24,500 | **3x more** |
| CPU usage | Baseline | -35% | **Significant** |

## When NOT to Use gRPC

gRPC isn't a silver bullet. Keep HTTP/REST for:

- **Public-facing APIs**: Browsers can't natively call gRPC (without gRPC-Web)
- **Simple CRUD services**: The overhead of proto definitions isn't justified
- **Third-party integrations**: Most external services speak REST
- **Debugging**: JSON is human-readable; protobuf is not

In G-NEXA, we maintain REST through **Kong API Gateway** for all external client communication, while internal service-to-service calls use gRPC exclusively.

## Conclusion

The migration from HTTP to gRPC in our microservices architecture was one of the highest-impact changes we made. The combination of binary serialization, HTTP/2 multiplexing, strong typing across languages, and built-in streaming made our inter-service communication significantly faster and more reliable.

If you're running a distributed system with more than 3-4 services, especially in a polyglot environment, gRPC should be a serious consideration for your internal communication layer.
